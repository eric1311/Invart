Kappaski Code Design And Development Standards

这份文档记录当前代码重构判断、模块边界、测试驱动规范和后续开发约束。目标是让 Kappaski 从“版本功能堆叠”稳定收口为可维护、可解释、可验证的 control-plane 产品。

当前重构结论

当前代码已经完成第二轮结构化收口：src/kappaski/cli.py 保留为兼容入口，命令解析和 handler 按领域进入 src/kappaski/commands/；commands/parser.py 只负责组装 parser，具体 argparse 注册拆到 parser_foundation.py、parser_governance.py、parser_integrations.py、parser_product.py；src/kappaski/evals.py 保留为 façade，benchmark runner 按 semantic、foundation、v0.8-v0.18、v0.19-v0.29、full-product、registry 等切片进入 src/kappaski/benchmarks/；原 tests/test_core.py 已拆成按产品边界组织的测试切片；JSON/HTML artifact 写入统一由 src/kappaski/artifacts.py 提供稳定格式、hash 和相对链接 helper。

问题本次处理后续规范
CLI parser / handler 过深cli.py 现在只负责 main() 和 parser 入口；commands/parser.py 注册参数，commands/foundation.py、governance.py、integrations.py、product.py 承载 handler。新增 CLI 必须用 set_defaults(handler=...) 注册到对应 command family，不再追加主函数 if-chain。
benchmark/eval façade 过重evals.py 只继续暴露 run_benchmark() 兼容接口；runner registry 和 suite 实现进入 benchmarks/。suite 名称保持稳定；新增 benchmark 必须进入 registry，并保留 CLI、docs、roadmap capability。
CLI argparse 注册仍集中commands/parser.py 进一步收口为 20 行左右的组合器，命令族注册分别进入 parser family modules。新增命令先选择 parser family，再选择 handler family；不要把 argparse 注册重新堆回主 parser。
benchmark runner 仍集中benchmarks/core.py 进一步收口为 façade，runner implementation 按版本段和产品场景拆分。新增 benchmark 应进入最接近的 runner module；registry 只暴露 suite-name 到 runner 的映射。
测试定位困难test_core.py 拆成 runtime foundation、governance/identity、integrations、policy/evidence/RC、demos/roadmap 等测试切片。新增测试按产品问题归档，避免重新形成单个巨大测试文件。
artifact 写入重复artifacts.py 统一稳定 JSON、HTML、sha256、相对链接 helper，并迁移 proof、gate、replay、graph、coverage、evidence、demo、experiment、RC 报告写入。ledger JSONL 仍由 ledger.py 作为事实源管理；其他 portable artifact 优先使用 helper。
benchmark dispatch 是长 if-chain改为 registry-style runner dispatch，新增 suite 只注册 runner。新增 benchmark 必须有 runner、registry entry、CLI test、docs。
真实 demo 数据散落在文档叙述中新增 real_world_cases.py，把公开来源、claim boundary、before/during/after 映射结构化。所有真实世界 claim 必须进入可导出的 JSON，而不是只写在 HTML 里。
文档和代码容易脱节新 demo 同时提供 CLI、benchmark、HTML 入口和测试。版本完成必须同步更新 docs index、user guide/briefing 或 roadmap。

推荐模块边界

CLI parser→Command handler→Domain service→Ledger / Artifact writer→Benchmark / Gate
层职责不要做什么
CLI parser只负责参数、帮助文本、基本枚举。不要直接拼装复杂业务对象或写 artifact。
Command handler把参数转成 domain service 调用，并打印 JSON。不要重复策略判断、风险判断、hash 逻辑。
Domain service实现 identity、policy、mediation、graph、evidence、demo 等产品能力。不要依赖 stdout 或 argparse。
Artifact writer稳定写 JSON/HTML/JSONL，保证路径、hash、schema 可复现。不要把 claim boundary 只藏在页面文案里。
Benchmark/Gate验证产品问题是否被回答，而不是只测函数返回。不要把 optional heavy skip 伪装成 external validation pass。

代码规范

Schema First

新增可导出对象必须有 schema_version，稳定 JSON 输出要使用 sort_keys=True。

Ledger First

ledger 是事实源；proof、replay、graph、audit、bundle 都是派生表示。

Coverage Honest

observed、mediated、enforced、fail_open_alert 不得混淆。

Claim Boundary

公开报告映射、模拟 fixture、contract test、official benchmark run 必须明确区分。

LLM Boundary

LLM reviewer 可升级风险和生成说明，但不能降级 deterministic critical rule。

HTML Docs

面向用户、架构、roadmap 和版本完成说明优先写 HTML，并补充图形化结构。

TDD 完成定义

每个新能力必须先回答产品问题，再谈实现细节。

1. 写失败测试：CLI、domain service、artifact、benchmark 至少覆盖两个层面。
2. 实现最小能力：不绕过 ledger、proof、coverage、claim boundary。
3. 补 benchmark：必须输出 metrics 或 checks，能被 eval benchmark 调用。
4. 补文档：HTML version/architecture/user-facing entry 至少更新一处。
5. 跑验证：pytest、对应 benchmark、roadmap/RC gate 或 targeted gate。
6. 不允许把 mock 结果当作真实外部验证；optional heavy 必须 clean skip。
