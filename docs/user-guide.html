Kappaski

User Guide, Quick Guide, And Tutorial

How to use the current Agent Runtime Control Plane capabilities through v0.40: preflight, sessions, runtime policy, approvals, daemon authority, adapters, proof, gate, replay, TeamRun, harness compatibility, Rust enforcement, native integration inventory, hook/plugin bridge, MCP broker, coverage-aware gates, identity binding, path graph, path-aware policy, unified mediation, profile governance, evidence bundles, release gates, benchmark-derived LLM agent experiments, and the full SWE-Bench validation contract.

Core Loop

Scan→Session→Invocation→Policy→Approval / Block→Ledger→Proof→Gate / Audit

Fastest Useful Demo

kappaski demo enterprise-audit --mode live-adapter --out-dir /tmp/kappaski-demo-live

Generates ledger, proof, replay, audit JSON, and audit HTML while blocking unsafe deletion through Rust file-write enforcement.

Pre-v1 Control Plane Demo

kappaski demo pre-v1-control-plane --out-dir /tmp/kappaski-pre-v1

Generates ledger, proof, replay, path graph, coverage report, audit report, and metrics.

Run A Command Safely

kappaski adapter run --target . --out-dir .kappaski/run --gate managed -- python3 -m pytest -q

Creates session artifacts and optionally gates them.

Block Unsafe Delete

kappaski enforce run-file-write --ledger ledger.jsonl --session ks_demo --target . -- sh -c "touch x; rm -rf ."

Returns 126 and blocks before execution.

10 Minute Tutorial

1. Preflight

kappaski pre-runtime --target . --save --preflight .kappaski/preflight.json

2. Start Session

kappaski session start --target . --agent codex --goal "demo" --session-id ks_demo --ledger .kappaski/ledger.jsonl --preflight .kappaski/preflight.json

3. Record Events

kappaski runtime record-event --session ks_demo --ledger .kappaski/ledger.jsonl --review auto --policy-mode managed --event '{"type":"file_read","path":"/repo/.env"}'
kappaski runtime record-event --session ks_demo --ledger .kappaski/ledger.jsonl --review auto --policy-mode managed --event '{"type":"network","url":"https://external.example/upload"}'

4. Approve Or Reject

kappaski approval list --ledger .kappaski/ledger.jsonl
kappaski approval approve --ledger .kappaski/ledger.jsonl --all --approver lcy --reason "trusted tutorial run"

5. Export And Verify

kappaski session close --ledger .kappaski/ledger.jsonl
kappaski proof export --ledger .kappaski/ledger.jsonl --out .kappaski/proof.json
kappaski proof verify --proof .kappaski/proof.json --ledger .kappaski/ledger.jsonl
kappaski gate verify --proof .kappaski/proof.json --ledger .kappaski/ledger.jsonl --mode managed --out .kappaski/gate-report.json
kappaski replay export --ledger .kappaski/ledger.jsonl --out .kappaski/replay.html --gate-mode managed

Containerized Verification

Use the container suite before claiming a version complete. It builds Dockerfile.test, mounts the current checkout into /workspace, and runs pytest, the v0.40 contract benchmark, full-product readiness, roadmap full status, and the RC gate. It also confirms that external validation remains incomplete until real upstream evidence is attached.

scripts/container-test.sh local

The optional full SWE-Bench path uses the same container entry point. It is release evidence only after the official all-instance run finishes and produces the complete report bundle.

scripts/container-test.sh swe-bench-full gold kappaski_full_20260601 .kappaski/swe-bench-full

Full SWE-Bench External Validation

Use external-validation swe-bench-full when we need complete external evidence. It defaults to SWE-bench/SWE-bench, split=test, all instances required, and 2294 expected instances. The equivalent princeton-nlp/SWE-bench id is accepted. It validates results/<run_id>.json plus results/<run_id>/instance_results.jsonl, and the swebench==4.1.0 shape gold.<run_id>.json plus per-instance logs/run_evaluation reports. The local v0.40-swe-bench-full-validation-contract benchmark verifies the Kappaski contract; it is not itself a completed upstream SWE-Bench run.

kappaski external-validation swe-bench-full \
  --python /path/to/swebench-python \
  --predictions-path /path/to/predictions.jsonl \
  --run-id kappaski_full_2026_05_31 \
  --work-dir .kappaski/swe-bench-full \
  --max-workers 12 \
  --timeout 1800 \
  --out .kappaski/swe-bench-full/full-validation.json

Boundaries

  • Kappaski observes and controls managed flows. A process that bypasses Kappaski is not guaranteed to be observed.
  • v0.13 enforcement is wrapper-level file-write enforcement, not kernel/OS-level interception.
  • If the compiled Rust shim is missing or platform-incompatible inside a container, Kappaski falls back to deterministic Python file-write guard evidence and marks the decision with fallback: true.
  • Process-group supervision and JSONL MCP stdio transcript capture are implemented; kernel/OS interception and full MCP subprocess lifecycle supervision remain explicit boundaries.
  • LLM reviewer live quality benchmarking is not run by default.
  • roadmap status --require-full should pass for the current local control-plane product scope.
  • roadmap status --require-external-validation should fail until the full SWE-Bench and other external validation artifacts are actually attached.
