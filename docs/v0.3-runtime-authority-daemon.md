# Kappaski v0.3 Runtime Authority Daemon

Date: 2026-05-27
Status: implementation design and first local authority slice

## Product Goal

v0.3 moves Kappaski from a CLI-only proof workflow toward an Agent Runtime
Control Plane.

The daemon is not just a background service. Its product role is runtime
authority:

```text
observe -> decide -> approve/block -> persist -> explain -> verify
```

It should let multiple agents and users run autonomously while Kappaski owns the
session registry, state transitions, ledger writes, approvals, and policy-facing
runtime facts.

## Why The Daemon Exists

The core reason is not convenience. The core reason is authority.

A multi-agent or multi-user control plane needs one component that can answer:

- which sessions exist;
- who/what created them;
- what state each session is in;
- where its ledger lives;
- whether it is active, paused, interrupted, stopped, or deleted;
- what approvals are pending or resolved;
- whether the session heartbeat is fresh;
- what the latest risk, decision, taint, and outcome state is;
- which component is allowed to write facts to the ledger.

Without this, Kappaski remains a useful proof CLI but not a control plane.

## v0.3 First Slice

The first implementation deliberately avoids a network daemon and external
service dependencies. It provides a daemon-compatible local authority layer:

- file-backed session registry;
- lock-protected state writes;
- lock-protected authority entry points for ledger writes;
- CLI commands under `kappaski daemon ...`;
- session lifecycle operations;
- record-event / approve / outcome through the authority;
- status and session inspection.

This is enough to settle the data model and CLI semantics before introducing a
resident process.

## Session Lifecycle

Supported states:

- `active`: session is running and can receive events;
- `paused`: session is intentionally paused;
- `interrupted`: session was interrupted and may be resumed;
- `stopped`: session is closed and should not receive new events;
- `deleted`: session is hidden from active workflows but retained in registry for audit.

Initial commands:

```bash
kappaski daemon init --target .
kappaski daemon status --target .
kappaski daemon session create --target . --agent codex --goal "..."
kappaski daemon session list --target .
kappaski daemon session show --target . --session <id>
kappaski daemon session pause --target . --session <id>
kappaski daemon session resume --target . --session <id>
kappaski daemon session interrupt --target . --session <id>
kappaski daemon session stop --target . --session <id>
kappaski daemon session delete --target . --session <id>
kappaski daemon record-event --target . --session <id> --event '<json>'
```

## Registry Record

Each registry session should include:

- `session_id`;
- `status`;
- `target`;
- `agent`;
- `goal`;
- `ledger_path`;
- `created_at`;
- `updated_at`;
- `started_at`;
- `stopped_at`;
- `last_heartbeat_at`;
- `last_activity_at`;
- `last_decision_id`;
- `last_invocation_id`;
- `last_risk`;
- `last_effect`;
- `last_approval_grade`;
- `pending_approvals`;
- `metadata`.

## Authority Boundaries

The local authority owns registry state and should become the preferred writer
for runtime facts. Direct v0.1/v0.2 CLI ledger writes remain available for
compatibility, but adapters should move to daemon commands.

For v0.3, the authority guarantees process-local file locking, not OS-wide
sandbox enforcement. It is a control-plane foundation, not yet a kernel-level or
network-level enforcement layer.

## Non-Goals

v0.3 does not include:

- UI;
- full MCP proxy;
- browser agent adapter;
- OS sandboxing;
- network egress enforcement;
- remote multi-tenant server;
- enterprise dashboard.

## Implemented In This Tree

Implemented module:

- `src/kappaski/daemon.py`

Implemented CLI:

```text
kappaski daemon init
kappaski daemon status
kappaski daemon session create|list|show|pause|resume|interrupt|stop|delete
kappaski daemon record-event
kappaski daemon heartbeat
kappaski daemon approve
kappaski daemon reject
kappaski daemon outcome
```

Implemented test coverage:

- authority registers sessions and records events;
- registry tracks latest decision, invocation, risk, effect, and pending approvals;
- paused sessions reject runtime event writes;
- resume restores writability;
- approval and outcome update registry state;
- CLI daemon session flow.

## Later Implementation Upgrade

After this slice is stable, the same authority semantics can move behind a
resident daemon process with one of these implementations:

- Python service for fastest iteration;
- Go service for simple static distribution and concurrency;
- Rust service for strongest low-level enforcement path.

The current recommendation is to keep policy/review/reporting in Python while
extracting the resident authority and enforcement shims later only when the
adapter surface requires it.
