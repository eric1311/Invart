Kappaski

Concepts And Glossary

这个文档解释 Kappaski 文档和代码中反复出现的概念、英文单词和缩写：它们为什么存在、在架构里属于哪一层、和 roadmap 哪个阶段有关，以及应该去哪里继续阅读。

How To Read This

Kappaski 的核心不是“再做一个 agent”，而是在 Claude Code、Codex、Gemini CLI、Cursor、OpenCode、OpenClaw、Hermes 等 agent 之外，提供统一的 runtime control plane。很多术语看起来像工程对象，其实是在回答企业安全和个人用户都会问的四个问题：

Who

谁用什么 principal、agent identity 和 credential boundary 发起了行为？

What

agent 读了什么、写了什么、调用了什么工具、产生了什么 artifact？

Why

为什么允许、审计、要求 approval、阻断或 fail-open？策略和证据在哪里？

How Strong

Kappaski 对这个行为只是 observed，还是 mediated，甚至 enforced？覆盖强度是否被诚实记录？

Project Spine

Pre-runtime Scan→ Session Registry→ Invocation→ Policy / Mediation→ Ledger→ Proof / Replay→ Gate / Audit

对应阅读：Architecture、Roadmap、Product Decisions、User Guide。

Product And Architecture Terms

TermMeaningWhy It ExistsReference
Agent Runtime Control PlaneKappaski 的产品定位：围绕已有 agent 的运行时控制平面。避免把 Kappaski 理解成 Claude Code/Codex 的替代品；它负责治理、审计、风控和证据。Product Decisions
Control Plane管理、记录、决策、审计、验证 agent 行为的层。把 agent 的行为从不可追踪的操作变成可治理的 runtime transaction。Architecture
Agent实际执行任务的应用或模型驱动工具，例如 Codex、Claude Code、Cursor。Kappaski 不直接替代 agent，而是在 agent 外部形成统一治理层。Plugin Review
Adapter把不同 agent、hook、wrapper、MCP/tool 调用转成 Kappaski 可理解事件的边界层。不同 agent 接口不一样，adapter 负责归一化，而不是拥有最终事实源。Adapters
Daemon本地 authority：session registry、single-writer ledger、approval state、policy API。多 agent、多 user 环境下必须有统一事实写入点和状态管理者。v0.3
Native Hook / Plugin / Extensionagent 产品自身提供的扩展机制。它们适合早期 intent、UX 和 agent-specific context，但不能单独完成完整安全闭环。Plugin Review
MCP Broker透明代理 MCP JSON-RPC tool call 的组件。记录工具调用证据、折叠 raw content、保持兼容性。v0.17
Wrapper / Proxy / Shim更靠近执行面的包裹、代理或本地拦截层。plugin/hook 不足以证明所有副作用；更强的执行边界需要 wrapper/proxy/shim。Architecture

Runtime Evidence Terms

TermMeaningWhy It ExistsReference
Session一次 agent 运行的治理容器，包含目标、agent、身份、状态和 ledger。所有行为必须落到某个 session，才能被回放、审计、关闭或中断。User Guide
Session Registrydaemon 维护的 session 状态表。多 agent/多 user 下需要知道每个 session 的创建、暂停、恢复、停止和上下文。v0.3
Invocation规范化后的运行时行为记录：command、file、network、MCP、skill、content 等。让不同来源的行为进入同一个 policy、ledger、proof 和 graph 体系。v0.1 Concepts
Outcome某个 decision 最终执行、阻断、跳过、覆盖、失败的结果。没有 outcome 就只能证明“曾经决策”，不能证明闭环是否真的完成。v0.1 Concepts
Ledgerappend-only JSONL 事实源，带 sequence、previous hash、entry hash。它是事实源。proof、graph、audit 都应从 ledger 派生，而不是反过来。Architecture
Proof从 ledger 派生的可携带摘要。用于 CI、审计和分享；它方便但不是事实源。v0.5
Replay面向人阅读的运行过程时间线。帮助用户复盘命令、approval、raw content 折叠证据和 gate findings。v0.7
Audit Report面向安全团队的风险、证据、流程合理性报告。把 ledger/proof/graph 转成企业安全团队能复审的材料。v0.14
Gate消费 proof 和 ledger 的 pass/warn/fail 决策。让 Kappaski 的证据进入 CI、PR、release 或企业 policy gate。v0.5
Artifact一次运行产生的 ledger、proof、replay、graph、coverage、audit 等文件。产品闭环必须留下可验证、可复现、可引用的结果。Full Product Readiness

Identity And Authorization Terms

TermMeaningWhy It ExistsReference
Principal可追责的人、用户、服务账号或组织身份。回答“谁授权了这个 agent 做这件事”。v0.19
AgentIdentity被声明或推断的 agent 身份，例如 agent id、vendor、adapter。区分“谁在操作”和“哪个 agent 在执行”。v0.19

CapabilityGrant对某类能力、资源或工具调用的授权记录。把 pre-runtime 扫描发现的能力转成可治理的授权事实。v0.4
SessionBindingprincipal、agent identity、credential boundary、grant 与 session 的绑定。保证 proof 可以回答 who / agent / grant / credential boundary。v0.19
TeamRun多用户、多 agent 的运行上下文。企业环境里 agent 间协作和 handoff 需要统一事实模型。v0.12

BlackboardTeamRun 内共享的上下文/事实记录。为多 agent 协作提供可追踪共享空间，而不是靠不可审计的聊天上下文。v0.12

Risk, Policy, And Mediation Terms

TermMeaningWhy It ExistsReference
Taint敏感或不可信上下文进入 session 后留下的传播标记。taint 本身不等于阻断；它让后续 write/network/deploy 等行为被路径策略识别。v0.1 Concepts
Target Deviationagent 偏离用户目标或项目边界的行为。很多风险不是单条命令危险，而是行为路径与目标不一致。Product Decisions
Deterministic Critical Rule规则化且高置信的 critical 风险判断。LLM 可以补充语义，但不能把 deterministic critical 降级。v0.8
LLM ReviewerOpenAI-compatible reviewer，用于语义分级、解释和风险升级。有些行为不能靠正则或静态规则判断，需要语义判断；但 LLM 不是 root of trust。v0.8
Policy Modeaudit、advisory、managed、ci 等运行模式。不同用户和企业阶段对阻断、审批、记录的要求不同。Product Decisions
Approval人或策略对 require-approval 决策的接受/拒绝记录。让高风险行为可以暂停、解释、采纳、拒绝，并写入 ledger。v0.7
MediationRequest / Decision / Outcome跨 command、file、network、env、MCP、native hook 的统一请求/决策/结果协议。避免每个 surface 都有一套不同语义，导致 proof 和 audit 无法统一解释。v0.22
Path Graph从 ledger 派生的执行路径图。把事件列表升级成因果链：谁读了 secret，如何 taint，最终流向哪个 sink。v0.20
Path-Aware Policy基于 source、trust、taint、grant、resource、sink、coverage 的路径策略。真实风险经常存在于路径中，不在单点事件里。v0.21
Coverage GradeKappaski 对某行为的实际控制强度：none、declared、observed、mediated、enforced。它不是风险分，而是诚实说明“我们到底看见/控制到了什么”。v0.18
Fail-open Alert无法强制执行时允许继续，但产生 critical alert。现实系统不能总是 fail-closed；但 proof 不能声称已 enforced。v0.22
Break-glass企业模式下带 reason、actor、scope、expiry、review state 的紧急覆盖记录。默认禁止 local override，但仍需为紧急流程留下审计路径。v0.23
Raw Content Policy控制 raw_content 展示为 visible、folded、truncated、redacted 或 hidden。审计需要证据，但敏感内容必须可控展示。v0.23

Common Confusions

Proof vs Ledger

Ledger 是事实源；Proof 是 portable summary。审计和 gate 最强模式应验证 proof 与 ledger 一致。

Risk vs Coverage

Risk 描述行为危险程度；Coverage 描述 Kappaski 是否实际观察、调解或执行了控制。

Plugin vs Control Plane

Plugin/hook 能提供早期事件和 UX，但不能单独证明所有副作用。control plane 还需要 daemon、ledger、wrapper、proxy、shim。

LLM vs Deterministic Rule

LLM 负责语义补充和说明，deterministic critical rule 是不可被 LLM 降级的安全底线。

Approval vs Deny

Approval 是暂停后允许人工或策略决策；Deny 是直接拒绝。两者都应进入 ledger。

Replay vs Audit

Replay 更像时间线回放；Audit 面向安全团队解释风险、策略、证据和流程合理性。
