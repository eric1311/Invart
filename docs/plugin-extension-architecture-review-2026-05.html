Kappaski

Plugin And Extension Architecture Review

2026-05 architecture correction after reviewing mainstream agent plugin, hook, extension, MCP, and sandbox surfaces.

Accepted correction: Kappaski should be plugin-assisted, hook-aware, daemon-owned, and enforcement-backed. Native plugins and hooks are first-class integration layers, but not the only trusted enforcement boundary or fact source.

Architecture Position

Agent Plugin / Hook / Extension→Kappaski Adapter→Runtime Daemon→Ledger / Policy / Approval→Wrapper / MCP Proxy / Rust Shim

Boundary Split

Native Plugin / Hook

Best for intent, context, UX, early decision points, and vendor-specific metadata.

Not enough for cross-agent consistency, tamper-evident facts, subprocess effects, and OS-level enforcement.

Wrapper / Proxy

Best for portable launch control, command mediation, MCP mediation, and common evidence.

Not enough for rich in-agent context or IDE-native UX.

Daemon

Best for session registry, single-writer ledger, policy API, approval state, proof, and replay.

Not enough for OS interception by itself.

Native Shim / Sandbox

Best for stronger file, network, process, and secret enforcement.

Not enough for product UX, semantic reasoning, and audit reporting.

Reprioritize MCP as a cross-agent substrate for tool-call audit, approval, redaction, and compatibility.

Add dimensional coverage grade, observed-by, enforced-by, degraded reason, bypass surface, and profile-driven residual-risk gates. Enterprise signoff is deferred beyond v0.18.

Product sentence: Kappaski is a full-lifecycle agent runtime control plane that uses native plugins and hooks for first-class integration, but relies on a daemon-owned ledger, policy API, wrappers, proxies, and native shims for durable governance, cross-agent consistency, and stronger enforcement.
