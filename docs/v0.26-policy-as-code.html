v0.26 Policy-as-Code v1

Kappaski-native TOML profiles make path-aware control rules configurable and testable.

Goal

Move deterministic path policy from built-in checks to validated profile rules while preserving the invariant that critical deterministic rules cannot be downgraded by LLM review.

Rule Shape

FieldMeaning
sourcesecret, external_instruction, tainted context, or trusted source predicate.
sinkexternal_network, ci_deploy_mutation, destructive_shell, or future sinks.
effectallow, audit, require_approval, or deny.
criticalMarks deterministic critical rules as non-downgradable.

Example

[[policy.rules]]
id = "deny_secret_egress"
source = "secret"
sink = "external_network"
effect = "deny"
critical = true

CLI

kappaski policy validate --profile enterprise-policy.toml
kappaski policy test --profile enterprise-policy.toml
kappaski policy check-path --ledger ledger.jsonl --profile enterprise-policy.toml --out path-policy.json

Tests

PYTHONPATH=src /usr/bin/python3 -m pytest -q tests/test_core.py::test_v026_policy_as_code_validates_and_applies_path_rules
PYTHONPATH=src python3 -m kappaski.cli eval benchmark --suite v0.26-policy-as-code

Boundary

TOML is the implemented policy language. Rego and Cedar remain future compatibility targets, not active backends.
