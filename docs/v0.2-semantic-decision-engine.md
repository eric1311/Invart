# Kappaski v0.2 Semantic Decision Engine

Date: 2026-05-26
Status: local implementation complete, production LLM provider integrations pending

## Goal

v0.2 upgrades Kappaski from deterministic runtime proof foundation to semantic
runtime decision engine.

The core product value is approval grading:

- low-risk actions should be automatically allowed or approved;
- medium-risk actions should be audited or escalated depending policy mode;
- high-risk actions should require human approval;
- deterministic critical rules cannot be downgraded by semantic review.

## Architecture

```text
Invocation
  -> Evidence Builder
  -> Rule Engine
  -> Semantic Reviewer
  -> Policy Merger
  -> PolicyEvaluation
  -> Decision / Approval Grade
  -> Outcome
  -> Ledger
  -> Proof / Verify
```

## Review Modes

Runtime should support review mode:

- `off`: no semantic reviewer, rules only.
- `auto`: call reviewer when trigger conditions exist.
- `always`: review every invocation.
- `required`: like auto, but reviewer failure escalates to approval.

Initial implementation should include a deterministic `HeuristicReviewer` so the
pipeline can be tested without network or model dependencies.

A real LLM reviewer can later implement the same interface.

## Review Triggers

In `auto` mode, trigger semantic review when any of these are true:

- invocation source is untrusted or unknown external content;
- invocation has `tainted_session`, `sensitive_read`, `external_instruction`, or `credential` taint tags;
- operation is network, file write, shell, Skill load, Git operation, approval, or MCP high-risk action;
- deterministic rules found high or critical risk;
- session goal exists and invocation appears goal-sensitive.

## Data Model

### ReviewFinding

```python
@dataclass
class ReviewFinding:
    source: str              # rule | semantic | llm | heuristic
    category: str
    risk: str                # info | low | medium | high | critical
    confidence: float | None
    title: str
    reason: str
    evidence_refs: list[dict[str, str]]
    recommended_effect: str | None
```

### SemanticReview

```python
@dataclass
class SemanticReview:
    review_id: str
    session_id: str
    invocation_id: str
    reviewer: str            # heuristic | llm
    model: str | None
    prompt_version: str | None
    input_refs: list[str]
    input_hash: str
    risk: str
    confidence: float
    categories: list[str]
    reason: str
    recommended_effect: str
    findings: list[ReviewFinding]
    reviewed_at: str
    warnings: list[str]
```

### PolicyEvaluation

```python
@dataclass
class PolicyEvaluation:
    evaluation_id: str
    session_id: str
    invocation_id: str
    policy_version: str
    policy_mode: str
    review_mode: str
    rule_findings: list[Finding]
    semantic_reviews: list[SemanticReview]
    taint_tags: list[str]
    capability_grant_id: str | None
    final_effect: str
    final_risk: str
    approval_grade: str       # auto_approve | audit | require_human | blocked
    reason: str
    decision_trace: list[str]
    requires_approval: bool
    evaluated_at: str
```

## Effects And Approval Grades

Effects:

- `allow`
- `audit_only`
- `require_approval`
- `deny`
- `redact`
- `quarantine`

Approval grades:

- `auto_approve`: no human needed.
- `audit`: allowed but important in proof.
- `require_human`: user approval required.
- `blocked`: policy denies execution.

## Policy Merger Rules

Initial merger rules:

1. Deterministic critical findings produce `deny` unless explicitly configured otherwise later.
2. Deterministic critical findings cannot be downgraded by semantic review.
3. Deterministic high findings produce at least `require_approval` unless semantic review and policy mode allow a downgrade to `audit_only` for known low-risk contexts.
4. Semantic high or critical risk can upgrade a low-risk rule result to `require_approval`.
5. Semantic medium risk defaults to `audit_only` in `audit/advisory`, and `require_approval` in `managed/ci` when tainted.
6. Tainted session plus outbound/write-like action produces at least `require_approval` unless semantic review confidently marks it low-risk and policy mode allows auto-approval.
7. Reviewer failure in `required` mode produces `require_approval`.
8. `ci` mode treats missing approvals and invalid proof as failure conditions.

## Heuristic Reviewer

The first implementation should detect:

- archive/compress plus outbound destination;
- base64 or encoded blob with outbound action;
- external instruction phrases like ignore instructions or new objective;
- suspicious Skill source or Skill instruction;
- high-risk MCP/tool naming patterns, but MCP proxy itself is not v0.2 priority;
- approval reason that is empty or too vague, later.

This reviewer is not a replacement for LLM. It is a stable local test double and
baseline semantic layer.

## Real LLM Reviewer Contract

A future LLM reviewer must:

- consume redacted evidence only;
- never receive raw secret content by default;
- return strict JSON;
- include model, prompt version, input hash, confidence, categories, reason;
- fail closed according to review mode;
- write warnings to ledger/proof when output is invalid.

## Ledger Changes

Action ledger entries should add:

- `reviews`: list of semantic review dicts;
- `evaluation`: policy evaluation dict;
- `decision`: final policy decision dict, derived from evaluation.

For v0.2 compatibility, existing `decision` remains present.

## Proof Changes

Proof report should add:

- `semantic_reviews`;
- `policy_evaluations`;
- `review_warnings`;
- `execution_outcomes`;
- decision trace summaries.

Proof+ledger verification should compare review/evaluation counts and decision
references.

## CLI Changes

Runtime commands gain:

```text
--review off|auto|always|required
--policy-mode audit|advisory|managed|ci
```

Defaults:

- `--review auto`
- `--policy-mode advisory`

Useful commands:

```text
kappaski runtime record-event --review auto --policy-mode advisory ...
kappaski policy explain --ledger <ledger> --decision <id>
kappaski review invocation --ledger <ledger> --invocation <id>
```

Implemented CLI inspection commands:

```text
kappaski policy explain --ledger <ledger> --decision <id>
kappaski policy explain --ledger <ledger> --invocation <id>
kappaski review invocation --ledger <ledger> --invocation <id>
kappaski runtime outcome --ledger <ledger> --decision <id> --status executed
```

`runtime outcome` records the observed post-decision result as an append-only
ledger entry. This is the v0.2 bridge from decision evidence to actual closed-loop
execution evidence.

## Acceptance Tests

Required v0.2 tests:

- reviewer off preserves v0.1 deterministic behavior;
- auto mode reviews tainted outbound action;
- low-risk reviewed action can be auto-approved;
- high-risk semantic finding requires human approval;
- deterministic critical rule cannot be downgraded;
- reviewer failure in required mode requires approval;
- ledger stores reviews and policy evaluation;
- proof export includes reviews and evaluations;
- proof+ledger verification catches mismatched review/evaluation counts;
- outcome records are persisted and exported in proof;
- policy explain links invocation, decision, review, evaluation, approval evidence, and outcomes;
- review inspection can retrieve reviews by invocation.
