Kappaski V0.1 Prototype Design

AI Coding Agent 多 Agent 管理平台与运行时安全原型设计

1. 最终结论

一句话定义：Kappaski V0.1 应该是一个面向 AI Coding Agent 的多 Agent 运行时管理平台， 其中安全、权限、审计、审批、回放是第一等能力；它不是单纯 scanner，也不是只给某个 Agent 做插件。

产品主线

从“Agent 审计工具”升级为“多 Agent runtime control plane”。用户看到的是团队运行、能力授权、审批、审计和回放。

执行主线

采用 wrapper/proxy/daemon 作为受管 runtime 的强制控制面；plugin/hook 只能作为 adapter 和体验增强。

工程主线

下一步不要继续堆 regex，而要先落 Session → Invocation → PolicyDecision → Evidence 这条证据链。

1.1 这次讨论收敛出的不可妥协点

问题最终判断原因
是否走插件形式？ 插件可以做入口，但不能作为安全边界。 不同 agent 的 hook/plugin 覆盖不一致，且可能被 bypass、配置绕过或无法覆盖 OS 层行为。
是否必须走 wrapper？ V0.1 主路径必须是 managed runtime：shell wrapper + minimal MCP proxy + local daemon。 Shell、MCP、Git、审批、capability grant、跨 agent handoff 都必须进入统一 daemon 裁决和 ledger。
产品是否只做安全工具？ 不。安全是核心能力，但产品外显应是多 Agent 管理平台。 用户最终需要管理 goal、role、capability、approval、handoff、replay，而不只是看风险列表。
V0.1 最大技术风险是什么？ 只做安全底座会不像多 Agent 产品；只做协作 UI 又没有可信控制边界。 V0.1 必须交付 TeamRun、Blackboard、Handoff、Approval 与 daemon enforcement 共同工作的纵向闭环。

2. 三轮讨论综合

原始交付过度依赖 KAP-3 的既有讨论，没有在 KAP-4 下完成独立收敛。本修订补充了 KAP-4 的三轮讨论： 第一轮独立判断、第二轮互相质疑、第三轮由 Gogo 收敛为本文结论。

第一轮：方向确认

Hewitt、Hopper、Redding 均认可三段式生命周期：pre-runtime 做准入扫描，runtime 做行为控制，post-runtime 做审计回放。

共同质疑是：当前实现更像“规则扫描器 + 事件日志器”，还不是“执行控制平面”。

第二轮：对象模型和执行边界

讨论从命名争论收敛到双层对象模型：产品层用 Goal / Capability / Action / Decision / Evidence / Replay，工程层用 Session / Actor / Adapter / Invocation / ResourceRef / PolicyDecision / ArtifactRef。

MCP proxy 很重要，但必须建立在统一 envelope 上，不能先做成孤立日志。

第三轮：多 Agent 管理平台视角

最终将目标升级为“多 Agent 产品管理平台”。安全字段必须覆盖跨 agent handoff、权限漂移、污染上下文传递、审批绕过、审计日志可信度、hook/plugin 绕过后的保底控制。

Redding 强调：没有 source / trust_level / taint_tags / capability_grant_id / seq / hash_chain / approval_id，系统无法成为可信审计平台。

2.1 分歧和裁决

分歧Hewitt 观点Hopper 观点Redding 观点裁决
产品抽象 以 Goal、Capability、Policy 解释 agent 行动边界。 以 Session、Invocation、Decision 保证可实现。 必须加入 source、taint、evidence，否则不能归因。 采用双层模型：产品语义 + 工程 schema。
实现顺序 先做纵向最小闭环，避免过早抽象。 先 schema/policy/store，再 wrapper，再 MCP proxy。 schema 第一版就必须包含安全字段。 Slice 1 固定 schema、policy、hash store。
hook/plugin 是管理台入口和协作 UX。 是 adapter 输入，不是 root of trust。 不能覆盖绕过和 OS 层行为。 adapter-rich UX，daemon-enforced runtime；不再使用“plugin-first”作为架构口径。

2.2 KAP-4 补充讨论的第三轮收敛

议题收敛结论落到 V0.1 的要求
Wrapper / Plugin 路线 Hopper、Hewitt、Redding 一致确认：plugin/hook 不能作为安全保底；V0.1 主路径是 wrapper/proxy/daemon。 文档和原型统一改为 plugin-assisted adapter, daemon-enforced control plane。
安全 P0 范围 Redding 提出的 syscall、进程树、网络监控是正确方向，但 Hopper 的范围压缩更适合 V0.1。 V0.1 声明为 managed runtime prototype：只保证通过 Kappaski 启动的受管 session；OS 级 EDR/沙箱进入 V0.2。
多 Agent 产品闭环 Hewitt 的 TeamRun / Blackboard / Handoff 是产品必要骨架，但 Redding 指出它们也是 prompt injection 和权限传播通道。 Blackboard 和 Handoff 必须受 daemon 裁决；confidence、taint、grant、approval 不允许由 agent 自报。
最终 V0.1 形态 不是最大安全底座，也不是纯协作看板，而是最小纵向闭环。 交付 daemon + shell wrapper + minimal MCP proxy + approval/grant + hash-chain ledger + TeamRun/Blackboard/Handoff/Approval UI。

3. 产品定位与边界

应该做什么

  • 多 Agent session / team run 管理
  • Agent、MCP、Skill、CI、repo 配置准入扫描
  • Capability grant 和 scoped permission 管理
  • Shell / MCP / 文件 / 网络 / Git / Issue 的运行时裁决
  • 审批流、风险告警、阻断、隔离、脱敏
  • 基于证据链的 replay ledger

V0.1 不应该做什么

  • 不做泛化 AI 安全平台
  • 不做完整 SIEM / EDR
  • 不依赖某一个 agent 的插件生态
  • 不先做复杂多级 RBAC
  • 不承诺内核级 syscall 拦截、进程树冻结或完整网络 egress 防护
  • 不把 LLM 内容审查作为唯一安全机制

3.1 目标用户和核心问题

AppSec想知道 agent 是否读密钥、改 CI、调用高危 MCP。
研发负责人想放开 AI agent，但需要权限边界和团队治理。
开发者想继续用 Codex、Claude Code、Cursor，不想被粗暴禁止。
平台团队想统一管理多 agent 后端、skill、MCP 和审计证据。

4. V0.1 产品功能范围

Pre-runtime 扫描 agent、MCP、Skill、repo、CI、敏感路径，生成 preflight baseline。
Launch 创建受管 TeamRun、goal、actor、capability grant，通过 wrapper/proxy/daemon 启动 agent。
Runtime Shell wrapper / minimal MCP proxy 产生 invocation，daemon 做 policy decision；file/network monitor 进入后续增强。
Approval 高风险动作进入审批、阻断、隔离或脱敏，所有决策写入 event store。
Post-runtime 生成 replay timeline、风险链路、PR Bot 摘要和审计报告。

MCP / Skill Supply Chain Scanner 扫描 MCP server command、tool description、Skill 脚本、外链下载、隐藏指令、权限变化。 MCP/Skill 风险评分、quarantine 建议。

TeamRun / Blackboard / Handoff 把多 Agent 协作建模为事件流：claim、evidence、open question、decision、handoff contract。 Blackboard view、handoff approval、grant delegation audit。

5. 产品架构

Agent UI / CLI / IDE / CI / 多 Agent 管理台
Native Hook / Plugin Adapter（可用则接入，只做体验、上下文和观察，不做安全保底）
Kappaski Backend Adapter Layer（Codex、Claude Code、Cursor、Cline、Windsurf、CI、自研 Agent）
Local Runtime Daemon：Session Registry + Policy Engine + Capability Grants + Approval + Event Store + Replay Builder
Enforcement Shims：Shell Wrapper + MCP Proxy + File Monitor + Network Monitor
OS Sandbox / Container / Egress Proxy（后续增强）

5.1 为什么是“统一 runtime + 多个 enforcement shim”

跨后端

不同 agent 的 hook 能力差异很大。统一 daemon 可以把 Codex、Claude Code、Cursor、CI agent 和自研 agent 归一成同一种 runtime envelope。

可审计

所有 action 都必须有 session、seq、decision、evidence、policy version 和 hash chain，才能支持事后回放和合规证明。

可演进

V0.1 先做 daemon、shell wrapper、minimal MCP proxy 和 store，后续加入 file/network monitor、OS sandbox，不推翻核心 schema。

6. 核心对象模型

产品层对象

  • Goal：本次人类目标或团队目标。
  • AgentRole：Planner、Implementer、Reviewer、RedTeamer 等角色。
  • Capability：shell、file、network、MCP write、Git push、issue update 等能力。
  • Policy：对能力和资源的裁决规则。
  • Action：agent 试图执行的行为。
  • Decision：允许、拒绝、审批、隔离、脱敏。
  • Evidence：支撑裁决的证据。
  • Replay：可解释回放。

工程层对象

  • Session / TeamRun：一次 agent run 或团队 run。
  • ActorRef：user、agent、process、MCP server、adapter。
  • Adapter：codex wrapper、claude hook、MCP proxy、CI adapter。
  • CapabilityGrant：作用域、时效、审批来源。
  • Invocation：统一动作 envelope。
  • ResourceRef：file、URL、command、MCP tool、issue、artifact。
  • PolicyDecision：结构化裁决。
  • ArtifactRef：摘要、diff、日志、审批记录。
  • BlackboardEntry：claim、evidence、open_question、decision 的事件派生视图。
  • HandoffContract：delegator、delegatee、goal_delta、context_refs、grant_delta、approval_policy。

6.1 Mandatory Runtime Envelope

字段必须性说明
session_id, event_id, seqP0支撑 session 隔离、去重、顺序回放。
prev_event_hash, event_hashP0形成 hash chain，检测日志篡改。
actor_id, actor_type, adapter_idP0回答谁、通过哪个 adapter 发起动作。
operation, capability_used, capability_grant_idP0回答用了什么权限，是否超出 grant。
resource_refs, input_refs, output_refsP0建立资源、上游影响和下游结果。
source, trust_level, taint_tagsP0区分用户、repo、外部网页、issue comment、MCP result、Skill instruction。
decision, policy_version, matched_rule_idsP0解释当时为什么允许、阻断或要求审批。
approval_id, approval_state, approval_by, approval_expiryP0审批流可以先简单，但 action/grant/handoff 的审批 ID 必须进入事件链。
handoff_id, delegator_session_id, delegated_context_taintP0多 Agent handoff 必须可审计，grant delegation 只能 restrict-only。
redaction_state, adapter_signatureP1证据脱敏和 adapter 身份可信度。

6.2 多 Agent 协作契约

对象字段V0.1 规则
AgentMessage from_actor、to_actor、intent、payload_ref、taint_tags、correlation_id 跨 trust domain 消息必须经过 daemon 准入，payload_ref 继承上游 taint。
BlackboardEntry type、source_event_id、publisher_trust_level、taint_depth、confidence confidence 由 daemon 根据 trust 和 taint 计算；agent 不能自报高置信度。
HandoffContract delegator、delegatee、goal_delta、context_refs、grant_delta、approval_policy grant_delta 只能缩小权限；扩权必须走独立 capability grant 审批。
EvaluatorSynthesis synthesis_event_id、input_entry_ids、weights、source_trust V0.1 不做自动事实合成，只保留 schema 方向；避免过早信任折叠。

7. 安全策略模型

V0.1 的策略不是“命中 regex 就阻断”，而是基于矩阵裁决：

decision = f(source, trust_level, taint_tags, capability_used, resource_refs, sensitivity, grant, policy_version)

7.1 V0.1 必须内置的硬规则

规则默认动作示例
Untrusted source + write/exec/network/push require_approval 外部 issue comment 诱导 agent 创建 gist、发 Slack、执行 shell。
Sensitive read + outbound deny / approval 读取 .env 后访问外部网络、MCP 写工具、Git push。
Capability usage 超出 grant deny read-only agent 试图执行 git push 或数据库写。
Unknown / unpinned MCP server quarantine MCP stdio command 未固定版本，或来源不明。
Tainted session handoff require_approval 低权限 agent 读取外部网页后把任务交给高权限 agent。
Handoff grant escalation deny handoff 只能传递 delegator 与 delegatee 已有 grant 的交集，不能隐式扩权。
Blackboard taint depth ≥ 3 require_approval 多跳传播的 claim 被用于写操作、外发或 Git push。
Approval fatigue escalate / deny 同一 session 多次请求同一高危 capability 审批。

7.2 多 Agent 管理平台的六类攻击面

跨 Agent Handoff

污染任务从低权限 agent 传给高权限 agent，需要 handoff_id、delegator、delegated_goal、taint 和 approval。

权限漂移

Skill/MCP 动态请求新能力，必须产生 capability_request，超出 grant 时审批或拒绝。

污染上下文传递

外部网页、issue、PDF、MCP result 的内容进入上下文后，后续动作继承 taint。

审批绕过

审批必须由 daemon 处理，不能由 agent 进程内部自审批；审批过期自动拒绝。

审计日志可信度

event store 只接受 daemon 写入，记录 seq、prev hash、event hash 和 daemon 时间。

Hook/Plugin 绕过

hook/plugin 可作为观察点，shell wrapper、MCP proxy、daemon 才是保底控制层。

10. 典型用户流程

流程 A：安全工程师首次扫描

  1. 安装 CLI。
  2. 执行 kappaski pre-runtime --target /repo。
  3. 生成 agent、MCP、Skill、sensitive path、CI 风险清单。
  4. 选择 policy template，生成 preflight baseline。

流程 B：开发者运行 Agent

  1. 执行 kappaski run --agent codex --target /repo -- codex。
  2. daemon 创建 session 和 capability grant。
  3. 普通动作静默记录，高危动作审批或阻断。
  4. 结束后生成 replay report。

流程 C：多 Agent 协作

  1. 用户创建 team run 和 goal。
  2. 为 Planner、Implementer、Reviewer 分配不同 grant。
  3. Blackboard 发布 claim/evidence/open question，daemon 写入 trust 和 taint。
  4. handoff 产生 HandoffContract，grant_delta 只能 restrict-only。
  5. tainted handoff 或 grant escalation 要求审批。

流程 D：事后复盘

  1. 安全团队打开 high-risk session。
  2. 查看 source/taint/action/decision/evidence 链路。
  3. 定位敏感读、外发、MCP 写、Git push。
  4. 导出 HTML 报告或 PR Bot 摘要。

12.1 V0.1 验收标准

  • 每个 session 有稳定 session_id、goal、policy_version、capability_grant。
  • 每个 runtime action 都落成 Invocation + PolicyDecision + Evidence。
  • Event store 有 seq 和 prev_event_hash。
  • 至少支持 shell wrapper 的 allow / deny / require_approval 三态裁决。
  • 至少支持一个 MCP 写类工具通过 proxy 进入 daemon 裁决。
  • 至少支持一个 HandoffContract demo，grant delegation 只能 restrict-only。
  • Blackboard entry 显示 source、publisher trust、taint depth，读取和使用受 policy 裁决。
  • Replay report 能解释一次高危动作的来源、原因、裁决和结果。
  • 插件或 UI 只作为入口，不能绕过 daemon 的 policy/store/replay。

13. 最终建议

Kappaski V0.1 的正确路线是：以多 Agent 管理平台为产品外观，以 local runtime daemon 为技术内核， 以 wrapper/proxy 为执行控制底线，以 append-only replay ledger 为安全证据基础。

下一步开发不要围绕“到底是 plugin 还是 wrapper”继续争论。产品上要做 adapter-rich 的多 Agent 管理体验，让用户在 UI 中管理团队、角色、授权、审批、Blackboard、Handoff 和回放； 架构上必须 daemon-first，让所有 hook、plugin、wrapper、proxy 都只是 adapter 或 enforcement shim；安全上必须 evidence-first，让每个动作都有来源、权限、裁决和证据。

如果这条主线成立，Kappaski 就不是一个“限制 agent 的工具”，而是让企业能安全放开 AI Coding Agent 的运行时基础设施。
