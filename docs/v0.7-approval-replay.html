Kappaski

v0.7 Approval Resolution And Replay

Close the loop after a gate failure: list missing approvals, resolve them, re-export proof, pass the gate, and produce a replay report.

v0.7 goal: convert gate failure into a user-operable resolution loop and a readable runtime audit report.

Fix Loop

Gate Fail→Approval Inbox→Approve / Reject→Re-export Proof→Gate Pass→Replay HTML

Implemented CLI

kappaski approval list --ledger ledger.jsonl --status missing
kappaski approval approve --ledger ledger.jsonl --all --approver lcy --reason "trusted corpus"
kappaski proof export --ledger ledger.jsonl --out proof.json
kappaski gate verify --ledger ledger.jsonl --proof proof.json --mode managed
kappaski replay export --ledger ledger.jsonl --out replay.html --case benchmarks/cases/swe-bench-lite/pinned_cases.json
kappaski eval benchmark --suite v0.7-approval-replay

Approval Semantics

FeatureDecision
approve --allAllowed in the open mode, requires a reason, and records one approval evidence event per missing decision.
rejectRecords explicit rejection evidence; managed and CI gates continue to fail.
Closed sessionNot required by default. Staged proof should remain visible during active runs to avoid partial blind risk.

Replay Report

Shows

Session summary, real case context, timeline, policy decisions, approval states, gate findings, and folded raw proof JSON.

Does Not Become

A source of truth. Replay is derived from ledger/proof and can be regenerated.

Real Case Fixture

The default v0.7 benchmark attaches pinned SWE-Bench Lite metadata for django__django-11001. It does not run full patch grading in default tests; it uses real coding-agent task metadata as replay/session context.
